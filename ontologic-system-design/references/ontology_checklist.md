# 온톨로지 검증 체크리스트 (Ontology Validation Checklist)

프로젝트의 온톨로지 성숙도를 정량적으로 평가하는 체크리스트.
각 항목에 대해 해당(1점) / 미해당(0점)으로 채점한다.

---

## A. 엔티티 완전성 (Entity Completeness)

- [ ] **A1**: 모든 핵심 도메인 개념이 명시적 클래스/타입으로 존재하는가?
  - 스캔: `class`, `dataclass`, `TypedDict`, `BaseModel` 검색 → 도메인 용어와 대조
- [ ] **A2**: 암묵적 개념(string으로만 표현되는 enum 등)이 명시화되었는가?
  - 스캔: 매직 스트링 검색 (`"pending"`, `"active"` 등 하드코딩된 상태값)
- [ ] **A3**: 엔티티 속성이 primitive만이 아닌 도메인 타입을 사용하는가?
  - 예: `user_id: str` → `user_id: UserId` (NewType 또는 값 객체)
- [ ] **A4**: Value Object와 Entity가 구분되는가?
  - Entity: 고유 식별자, 생명주기 / Value Object: 불변, 동등성 비교

## B. 관계 명시성 (Relationship Explicitness)

- [ ] **B1**: 엔티티 간 관계가 암묵적 추론이 아닌 명시적 선언인가?
  - 스캔: ORM `relationship()`, FK 정의, 타입 참조 확인
- [ ] **B2**: 관계 타입이 구분되는가? (is-a, has-a, triggers, depends-on, observes)
  - 스캔: 관계 이름/변수명이 타입을 반영하는지 확인
- [ ] **B3**: 양방향 관계의 일관성이 보장되는가?
  - 예: A→B 관계 존재 시 B→A 역참조 가능
- [ ] **B4**: 관계 카디널리티(1:1, 1:N, M:N)가 코드에 명시되는가?

## C. 공리 / 불변식 (Axioms / Invariants)

- [ ] **C1**: 비즈니스 규칙이 코드 로직이 아닌 선언적 형태로 존재하는가?
  - 예: 상태 전이 테이블, 검증 규칙 딕셔너리, 설정 파일
- [ ] **C2**: 상태 전이가 명시적 상태 머신으로 구현되는가?
  - 스캔: `transitions`, `state_machine`, 또는 전이 딕셔너리 검색
- [ ] **C3**: 유효성 검증이 엔티티 레벨에서 통합 관리되는가?
  - 스캔: `validate()`, `@validator`, `check_invariant` 패턴
- [ ] **C4**: 불변식 위반 시 명확한 에러 경로가 존재하는가?
  - 스캔: 커스텀 예외 클래스, 도메인 에러 타입

## D. 메타-에지 (Meta-Edges)

- [ ] **D1**: 관계 생성 규칙이 관계 자체와 분리되어 정의되는가?
  - 예: "signal_strength > threshold이면 ALERT 관계 생성" 규칙이 설정으로 관리
- [ ] **D2**: 조건부 관계의 규칙이 하드코딩이 아닌 설정으로 관리되는가?
  - 스캔: threshold, condition, rule을 포함하는 config 파일
- [ ] **D3**: 관계 생성/삭제의 이벤트가 추적 가능한가?
  - 스캔: 관계 변경 시 이벤트 발행 또는 로그 기록

## E. 레버 / 메타-레버 (Lever / Meta-Lever)

- [ ] **E1**: 파이프라인이 명시적으로 정의되어 있는가?
  - 스캔: `pipeline`, `stage`, `step`, `phase` 키워드
- [ ] **E2**: 파이프라인 오케스트레이션(메타-레버)이 개별 레버와 분리되어 있는가?
  - 예: 개별 처리 로직 vs. 전체 실행 순서/재시도/폴백 관리
- [ ] **E3**: 파이프라인 실행 결과(성공/실패/소요시간)가 추적 가능한가?
  - 스캔: 로그, 메트릭 수집, 트레이스 ID
- [ ] **E4**: 파이프라인 간 의존성이 명시적인가?
  - 스캔: DAG 정의, 의존성 선언, 순서 보장 메커니즘

## F. 액티브 메타데이터 (Active Metadata)

- [ ] **F1**: 메타데이터가 단순 기록이 아닌 행동 트리거로 사용되는 곳이 있는가?
  - 예: 임계치 변경 → 자동으로 모니터링 규칙 갱신
- [ ] **F2**: 스키마/설정 변경이 자동으로 다운스트림에 전파되는가?
  - 예: 엔티티 정의 변경 → 파이프라인 필터 자동 갱신
- [ ] **F3**: 모니터링 임계치가 메타데이터에서 동적으로 로드되는가?
  - 스캔: 런타임 설정 로드, hot-reload, dynamic config

## G. 검색 구조 정합성 (Search Structure Alignment)

- [ ] **G1**: 온톨로지가 검색/탐색 경로와 일치하는가?
  - "온톨로지 = 검색 구조 = 접근제어 구조" 원칙
- [ ] **G2**: 코드 내 "find/search/query" 패턴이 온톨로지 관계를 따르는가?
  - 스캔: 검색 함수가 엔티티 관계를 통해 탐색하는지 확인
- [ ] **G3**: 접근 제어가 온톨로지 관계 기반인가? (ReBAC 원칙)
  - 스캔: 권한 확인이 역할 테이블이 아닌 관계 경로 탐색인지

## H. 교차공간 패턴 (Cross-Space Patterns)

- [ ] **H1**: 3개 이상 분석공간에 걸친 엔티티가 식별되고 문서화되었는가?
- [ ] **H2**: 교차공간 엔티티에 대한 통합 뷰/인터페이스가 존재하는가?
  - 예: 대시보드, 통합 API, 종합 보고서

## 점수 산정

**총점**: 체크 항목 수 / 27

### 성숙도 레벨 매핑

| 점수 | 레벨 | 이름 | 설명 |
|------|------|------|------|
| 0-7 | **Level 0** | 암묵적 온톨로지 | 엔티티·관계가 코드에 흩어져 있고 명시적 구조 없음 |
| 8-13 | **Level 1** | 부분적 명시화 | 엔티티 카탈로그 존재, 일부 관계 정의됨 |
| 14-19 | **Level 2** | 구조적 온톨로지 | 관계 타입 구분, 공리 정의, 파이프라인 명시 |
| 20-23 | **Level 3** | 메타 온톨로지 | 메타-에지, 레버/메타-레버, 자기 참조 구조 확립 |
| 24-27 | **Level 4** | 액티브 온톨로지 | 액티브 메타데이터, 교차공간 통합, 자기 개선 |

### 카테고리별 우선순위 가이드

- **A+B가 낮으면**: 먼저 엔티티와 관계를 명시화 (기초)
- **C가 낮으면**: 비즈니스 규칙을 선언적으로 추출 (안정성)
- **D+E가 낮으면**: 메타-에지와 파이프라인 정의 (확장성)
- **F+G가 낮으면**: 메타데이터 활성화와 검색 정합 (자동화)
- **H가 낮으면**: 교차공간 허브 엔티티 식별 (레버리지)
