# CMR (Capability Maturity Registry) 추적 템플릿

역량(Capability)의 생명주기를 추적하는 레지스트리. 시스템의 각 기능/모듈이 얼마나 성숙한지를 정량적으로 관리한다.

---

## CMR의 5가지 역할

1. **Binding Completeness**: 설계된 역량이 실제로 구현되었는가?
2. **Production Readiness**: 운영 환경에 배포할 수 있는 수준인가?
3. **Risk Management**: 이 역량의 변경이 다른 역량에 미치는 영향은?
4. **Maturity Tracking**: 시간에 따른 성숙도 변화 추적
5. **Operational Transparency**: 현재 문제 상황과 병목 파악

---

## 5단계 성숙도 정의

| 레벨 | 이름 | 설명 | 지표 기준 |
|------|------|------|-----------|
| 1 | **Initial** | 기능이 존재하지만 비구조적, 재현 불가능 | 코드 존재, 테스트 없음, 문서 없음 |
| 2 | **Managed** | 기본 프로세스 정의, 결과가 어느 정도 예측 가능 | 기본 테스트 존재, 에러 핸들링 일부 |
| 3 | **Defined** | 표준화된 프로세스, 조직 전체에서 일관 | 테스트 커버리지 60%+, 문서화, CI 통합 |
| 4 | **Measured** | 정량적 측정 가능, 데이터 기반 관리 | 메트릭 수집, SLA 정의, 성능 대시보드 |
| 5 | **Optimizing** | 지속적 개선, 자동 최적화 | 자동 스케일링, 자기 치유, A/B 테스트 |

## CMR 개별 등록 템플릿

```markdown

- **역량 ID**: CMR-XXXX
- **역량명**: [이름]
- **도메인**: [소속 서브시스템]
- **소유자**: [담당자/팀]
- **등록일**: YYYY-MM-DD
- **현재 레벨**: X/5

### 설명
[이 역량이 하는 일을 1-2문장으로]

### 관련 파일
- `path/to/main_module.py`
- `path/to/config.yaml`

### 관련 온톨로지 엔티티
- [어떤 엔티티와 관련되는지]

### 관련 분석공간
- [6공간 중 어디에 해당하는지, 교차공간 점수]

### 성숙도 이력

| 날짜 | 레벨 | 변경 사유 | 근거 |
|------|------|-----------|------|
| YYYY-MM-DD | 1 | 초기 등록 | 기능 존재 확인 |
| YYYY-MM-DD | 2 | 테스트 추가 | 단위 테스트 8건, 에러 핸들링 |
| YYYY-MM-DD | 3 | 문서화+CI | coverage 72%, API 문서 |

### 다음 레벨 달성 조건
- [ ] 조건 1: [구체적, 측정 가능한 기준]
- [ ] 조건 2: [...]
- [ ] 조건 3: [...]

### 의존성
- **선행 역량** (이것이 의존하는): CMR-YYYY, CMR-ZZZZ
- **후행 역량** (이것에 의존하는): CMR-AAAA

### 메트릭

| 측정 항목 | 현재 값 | 목표 값 | 추세 |
|-----------|---------|---------|------|
| 성공률 | 98.2% | 99.5% | ↑ |
| 평균 지연 | 340ms | 200ms | → |
| 테스트 커버리지 | 65% | 80% | ↑ |
```

## 프로젝트 수준 CMR 대시보드 템플릿

> **참고**: 이 대시보드는 CMR 레지스트리가 구축된 이후 운영 단계에서 사용한다. 초기 온톨로지 분석 시에는 위의 개별 등록 템플릿만 사용.

```markdown

### 전체 건강도
- **총 역량 수**: XX개
- **평균 성숙도**: X.X/5
- **SLA 준수율**: XX% (목표: 95%)

### 성숙도 분포

| 레벨 | 역량 수 | 비율 | 바 |
|------|---------|------|----|
| 5 - Optimizing | X | XX% | ████ |
| 4 - Measured | X | XX% | ████████ |
| 3 - Defined | X | XX% | ██████████ |
| 2 - Managed | X | XX% | ██████ |
| 1 - Initial | X | XX% | ███ |

### 핵심 역량 현황 (교차공간 점수 8+ 우선)

| ID | 역량명 | 레벨 | 교차 점수 | 추세 | 다음 목표 |
|----|--------|------|----------|------|-----------|
| CMR-0001 | [Name] | 3 | 13/15 | ↑ | L4 |
| CMR-0002 | [Name] | 2 | 10/15 | → | L3 |

### 위험 역량 (Level 1-2 이면서 교차공간 점수 8+)
[교차공간 허브이면서 성숙도가 낮은 역량 = 최우선 개선 대상]

| ID | 역량명 | 레벨 | 교차 점수 | 위험 요인 |
|----|--------|------|----------|-----------|
| CMR-0005 | [Name] | 1 | 11/15 | 테스트 없음, 문서 없음 |

### 레벨 전환 이력 (최근 30일)

| 날짜 | 역량 | 변경 | 사유 |
|------|------|------|------|
| MM-DD | CMR-0003 | 2→3 | CI 통합 완료 |
| MM-DD | CMR-0007 | 3→4 | 메트릭 대시보드 구축 |
```

## CMR 운영 가이드

### 언제 CMR에 등록하는가?
- 새로운 기능/모듈이 Phase 1 분석에서 엔티티로 식별될 때
- 기존 기능이 리팩토링/확장될 때
- 장애/인시던트 발생 후 해당 역량 검토 시

### 갱신 주기
- **Level 1-2**: 주 1회 점검
- **Level 3**: 격주 점검
- **Level 4-5**: 월 1회 점검 (자동 메트릭 기반)

### 교차공간 역량 우선 관리
교차공간 점수 8+ 역량은 시스템의 핵심 허브. 이들의 장애는 전체 시스템에 파급.
- 이 역량들은 Level 3 이상을 최소 목표로 설정
- Level 2 이하인 교차공간 역량은 **위험 역량**으로 분류하여 즉시 개선
