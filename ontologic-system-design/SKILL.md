# Ontologic System Design

프로젝트의 아키텍처를 온톨로지 관점에서 분석하고, 확률적 지능(암묵적 패턴)에서 구조적 지능(명시적 그래프 기반 추론)으로 개선하는 스킬.

> **SOS 철학**: "가장 본질로, 가장 물리적으로.. 지능을 해체해 들어가는 것만이 AI의 내일입니다."

---

## 핵심 원칙 3가지

### 1. "Search before code"
코딩 전에 기존 온톨로지(엔티티, 관계, 규칙)를 먼저 검색하라. 새 코드를 작성하기 전에 기존 구조를 이해하고 재사용하는 것이 온톨로지의 복리 축적을 가능하게 한다.

### 2. "온톨로지 = 검색 구조 = 접근제어 구조"
하나의 온톨로지가 세 가지 역할을 동시에 수행한다:
- **지식 구조**: 개념 간 관계를 정의
- **검색 구조**: 정보를 찾는 경로를 정의
- **접근제어 구조**: 누가 무엇에 접근할 수 있는지 정의 (ReBAC)

### 3. 메타-에지: "관계를 정의하는 규칙"
데이터 간 연결이 아니라, 연결을 만드는 기준 자체. 메타-에지 하나를 변경하면 하위의 모든 일반 에지가 영향을 받는다 — 이것이 레버리지.

---

## 5-Tier 컨텍스트 매핑

프로젝트를 분석할 때, 먼저 현재 위치를 파악한다:

| Tier | 이름 | 해당하면 | 코드 지표 |
|------|------|---------|-----------|
| **-1** | 순수기초 | DB, 기본 ML 사용 | SQLite/Postgres, numpy, pandas |
| **0** | 응용기초 | 임베딩, RAG, 지식그래프 사용 | ChromaDB, LangChain, Neo4j |
| **1** | 핵심 | 명시적 온톨로지, GraphRAG 존재 | OWL/RDF, 엔티티 그래프, 커뮤니티 탐지 |
| **2** | 프레임워크 | 분석공간, ReBAC, 메타온톨로지 적용 | 6공간 매핑, 관계 기반 접근제어 |
| **3** | 구현 | NOMIK, 레버/메타-레버, 물리엔진 | 코드→그래프, 파이프라인 오케스트레이션, 프랙탈 분해 |

대부분의 프로젝트는 Tier -1 ~ 0에 위치한다. 이 스킬은 현재 Tier에서 한 단계 위로 올리는 구체적 방법을 제시한다.

### 성숙도 체계 참고

이 스킬은 두 가지 성숙도 체계를 사용한다:

| 체계 | 범위 | 용도 | 참조 파일 |
|------|------|------|-----------|
| **온톨로지 성숙도** | Level 0~4 | 프로젝트 전체의 구조적 성숙도 | `ontology_checklist.md` |
| **역량 성숙도 (CMR)** | Level 1~5 | 개별 기능/모듈의 운영 성숙도 | `cmr_template.md` |

온톨로지 성숙도는 "구조가 얼마나 명시적인가?", CMR은 "기능이 얼마나 안정적인가?"를 측정한다. 두 체계는 보완 관계.

## 실행 워크플로: 4단계

스킬이 호출되면 아래 4단계를 순서대로 진행한다. 각 Phase 종료 시 요약을 출력하고 다음 Phase 진행 여부를 사용자에게 확인받는다.

**Phase 전환 프로토콜**:
- Phase 1 종료 시: 6공간 진단 테이블 + 성숙도 점수 출력 → "Phase 2(온톨로지 설계)로 진행할까요?"
- Phase 2 종료 시: 설계된 온톨로지 요약 출력 → "Phase 3(개선 권고)로 진행할까요?"
- Phase 3 종료 시: 권고 테이블 출력 → "Phase 4(구현 가이드)로 진행할까요?"

### Phase 1: 프로젝트 온톨로지 분석

**목표**: 코드베이스에 암묵적으로 존재하는 온톨로지를 명시적으로 드러내기

**Step 1.1 — 코드베이스 스캔**

다음 패턴으로 빠른 초기 스캔을 수행한다. 공간별 심층 스캔은 `references/analysis_spaces.md`의 '스캔 방법'을 참조:

```
[엔티티 식별]
Glob: "src/**/*.py", "**/*.ts", "**/models/**", "**/schemas/**", "**/*.go"
Python:  Grep: "class \w+|dataclass|BaseModel|TypedDict"
JS/TS:   Grep: "interface \w+|type \w+ =|export class|@Entity|@Schema"
Go:      Grep: "type \w+ struct"
→ 핵심 도메인 클래스 목록 추출

[관계 식별]
Python:  Grep: "ForeignKey|relationship|has_many|belongs_to"
JS/TS:   Grep: "@ManyToOne|@OneToMany|@JoinColumn|ref:|populate\("
공통:    Grep: "^from .* import|^import " → import 의존성 그래프
이벤트:  Grep: "emit\(|publish\(|subscribe\(|on\(|addEventListener"
→ 엔티티 간 관계 맵 구축

[규칙/공리 식별]
Grep: "if .*status|state.*==|assert |validate|@validator"
Grep: "threshold|limit|max_|min_|policy|rule"
Grep: "class.*Enum|CHOICES|STATUS_|enum \w+"
→ 비즈니스 규칙과 불변식 목록
```

**Step 1.2 — 6공간 매핑**

발견된 각 엔티티를 `references/analysis_spaces.md`의 프레임워크에 따라 6개 공간에서 점수화한다.

출력: 진단 템플릿 테이블 (엔티티 × 6공간 점수)

**Step 1.3 — 온톨로지 성숙도 판정**

`references/ontology_checklist.md`의 25개 항목을 코드 스캔 결과와 대조하여 점수를 산정한다.

출력:
- 총점 / 25
- 성숙도 레벨 (Level 0~4)
- 카테고리별 점수 분포
- 가장 낮은 카테고리 = 최우선 개선 영역

---

### Phase 2: 온톨로지 설계

**목표**: 분석 결과를 바탕으로 개선된 온톨로지를 설계

**Step 2.1 — 범위 설정**
- 전체 프로젝트가 아닌, Phase 1에서 교차공간 점수가 가장 높은 서브시스템 1개를 선택
- "70% 완성도로 시작" — 완벽한 온톨로지는 없다

**Step 2.2 — 도메인 온톨로지 구조화**

4가지 구성 요소를 정의한다:

```
[클래스] 핵심 엔티티

  모니터링:  Signal, Alert, Case, Rule, Report
  이커머스:  Customer, Order, Product, Cart, Payment

[속성] 엔티티 특성

  모니터링:  Alert.severity, Signal.z_score
  이커머스:  Customer.tier, Order.total_amount

[관계] 엔티티 간 연결 (타입 명시)

  모니터링:  Signal —[triggers]→ Alert —[escalates_to]→ Case
  이커머스:  Customer —[places]→ Order —[contains]→ Product

[공리/규칙] 제약 조건

  모니터링:  "Alert는 RESOLVED 외에는 역방향 전이 불가"
  이커머스:  "VIP 고객 50만원 이상 무료배송"
```

**Step 2.3 — 메타-에지 식별**

관계를 만드는 규칙을 분리:
- 어떤 조건에서 관계가 생성되는가?
- 그 조건은 어디에 정의되어 있는가? (하드코딩 vs 설정)
- 조건 변경 시 영향 범위는?

**Step 2.4 — 레버/메타-레버 구조**

기존 파이프라인을 식별하고 오케스트레이션 구조를 매핑:

```
[레버 (개별 파이프라인)]
  예: 데이터 수집 → 신호 계산 → 임계치 판정

[메타-레버 (파이프라인의 파이프라인)]
  예: 스케줄러가 위 파이프라인을 주기적 실행 + 재시도 + 결과 추적
```

5가지 속성으로 레버 품질 평가:
- 합성성: 작은 단위로 조합 가능?
- 관찰 가능성: 각 단계 상태 추적 가능?
- 멱등성: 동일 입력 → 동일 결과?
- 실패 복원력: 자동 재시도/폴백?
- 버전 관리: 변경 이력 보존?

**Step 2.5 — 벡터 의미 강화 기회** (해당 시)

시멘틱 검색/임베딩이 적용된 프로젝트에서:
1. 유사단어 확장 (동의어, 관련어)
2. 의미 다층화 (추상→구체 계층)
3. 도메인 온톨로지 정렬 (의미 드리프트 방지 앵커)

---

### Phase 3: 개선 권고

**목표**: 구체적이고 우선순위가 명확한 개선 사항 도출

**Step 3.1 — 갭 분석**

Phase 1 점수를 기반으로 체크리스트 미충족 항목을 분류:
- **즉시 개선** (A+B 카테고리): 엔티티와 관계 명시화
- **안정성 강화** (C 카테고리): 규칙/불변식 선언화
- **확장성 확보** (D+E 카테고리): 메타-에지, 파이프라인
- **자동화** (F+G 카테고리): 액티브 메타데이터
- **레버리지** (H 카테고리): 교차공간 통합

**Step 3.2 — 파일 수준 권고**

각 권고를 구체적 파일 경로와 변경 내용으로 명시:

```markdown
### 권고 #1: [제목]
- **카테고리**: [A-H]
- **우선순위**: 높음/중간/낮음
- **대상 파일**: `src/models/alert.py`
- **현재 상태**: 상태 전이가 if-else 체인으로 구현
- **권고 내용**: 상태 전이 테이블을 딕셔너리로 분리하여 선언적 관리
- **영향 범위**: alert_center.py, surveillance.py
- **노력 수준**: 소 (1-2시간)
```

**Step 3.3 — CMR 추적 제안**

`references/cmr_template.md`를 기반으로:
- 교차공간 점수 8+ 엔티티를 CMR 역량으로 등록 제안
- 각 역량의 현재 성숙도 레벨 평가
- 다음 레벨 달성 조건 정의

**Step 3.4 — 액티브 메타데이터 기회**

수동 메타데이터 → 액티브 메타데이터 전환 가능 지점 식별:
- 설정 변경 시 자동 전파되어야 할 곳
- 임계치가 동적으로 로드되어야 할 곳
- 스키마 변경이 다운스트림에 자동 반영되어야 할 곳

---

### Phase 4: 구현 가이드

**목표**: Phase 3 권고를 실행 가능한 순서로 정리

**Step 4.1 — 우선순위 액션 아이템**

의존성을 고려한 위상 정렬:
```markdown
1. [선행 조건 없음] 엔티티 타입 명시화 (A1, A2)
2. [1 완료 후] 관계 타입 정의 (B1, B2)
3. [2 완료 후] 상태 머신 선언화 (C1, C2)
4. [3 완료 후] 메타-에지 분리 (D1, D2)
5. [병렬 가능] 파이프라인 관찰 가능성 (E3)
```

**Step 4.2 — 검증 체크리스트**

각 액션 아이템 완료 후:
- [ ] 기존 테스트 통과 확인
- [ ] 온톨로지 체크리스트 재채점 → 점수 상승 확인
- [ ] 영향 범위 내 모듈 동작 확인
- [ ] 교차공간 엔티티의 6공간 점수 변화 확인

## 출력 보고서 형식

최종 출력은 `# 온톨로지 시스템 분석: [프로젝트명]` 제목 아래 Phase 1~4 결과를 순서대로 합친 보고서로 구성한다. 각 Phase의 출력은 해당 Phase에서 정의한 형식을 따른다. 6공간 진단 테이블은 `references/analysis_spaces.md`의 진단 템플릿을 사용한다.

## 프로젝트 유형별 분석 포인트

### Web API / 백엔드
- **핵심 엔티티**: User, Resource, Permission, Session
- **주요 관계**: 인증/인가 경로, CRUD 연결
- **우선 공간**: 구조(API 토폴로지), 인과(요청→응답 체인)
- **레버**: 미들웨어 파이프라인, 요청 처리 체인

### 데이터 파이프라인 / ETL
- **핵심 엔티티**: Source, Transform, Sink, Schema
- **주요 관계**: 데이터 흐름 엣지, 스키마 의존성
- **우선 공간**: 시간(배치 주기), 인과(실패 전파)
- **레버**: ETL 스테이지, 오케스트레이터

### 모니터링 / 서베일런스 시스템
- **핵심 엔티티**: Signal, Alert, Case, Rule, Report
- **주요 관계**: 신호→경보 트리거, 상태 전이
- **우선 공간**: 시간(상태 머신), 재귀(규칙이 규칙 생성), 인과(에스컬레이션)
- **레버**: 감시 파이프라인, 보고서 생성 파이프라인

### 프론트엔드 / SPA
- **핵심 엔티티**: Component, State, Route, Event
- **주요 관계**: 컴포넌트 트리, 상태 흐름, 라우트 매핑
- **우선 공간**: 계층(컴포넌트 트리), 구조(상태 관리)
- **레버**: 렌더링 파이프라인, 데이터 페칭 체인

### ML / AI 시스템
- **핵심 엔티티**: Model, Dataset, Feature, Experiment, Metric
- **주요 관계**: 학습 데이터→모델, 피처 의존성
- **우선 공간**: 시간(학습 이력), 재귀(하이퍼파라미터 튜닝), 인과(피처 영향)
- **레버**: 학습 파이프라인, 평가 파이프라인

## 워크플로 가이드라인

1. **분석 전 수정 금지**: Phase 1~3은 읽기 전용. Phase 4에서만 변경 권고
2. **범위 제한**: 대규모 프로젝트는 서브시스템 단위로 분석. 한 번에 전체 불가
3. **70% 원칙**: 완벽한 온톨로지를 추구하지 않는다. 반복 정제가 핵심
4. **교차공간 우선**: 교차공간 점수가 높은 엔티티부터 개선
5. **선언적 전환**: 코드 로직을 선언적 구조(설정, 테이블, 스키마)로 추출하는 것이 핵심 개선 방향
6. **복리 축적**: 도메인 온톨로지 → 메타-온톨로지(공통 패턴) → Universal Principles로 축적
7. **컴팩트 모드**: 소규모 프로젝트(엔티티 10개 미만 또는 Tier -1)에서는 Phase 1~4를 단일 패스로 실행하고 마지막에 한 번만 확인
